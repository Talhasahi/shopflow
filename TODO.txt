SHOPFLOW - STEP-BY-STEP LEARNING ROADMAP

IDEA
ShopFlow is a multi-vendor marketplace:
- CUSTOMER browses approved products, uses a cart, and places orders.
- VENDOR creates products and receives orders.
- ADMIN approves/rejects products and manages the marketplace.

Stack: Next.js App Router, TypeScript, Tailwind, shadcn/ui, Prisma, MySQL,
Auth.js, Stripe, background jobs, realtime notifications, and AI.

RULES FOR LEARNING
1. Work on one checkbox at a time.
2. Ask: "Build this task and explain it like I am a Laravel developer."
3. Run and test the change yourself.
4. Mark it [x], make a Git commit, then continue.
5. Test every CHECKPOINT before starting the next phase.
6. Add packages only when their phase needs them.

PHASE 9 - BACKGROUND JOBS (LARAVEL QUEUE EQUIVALENT)
============================================================
Next.js has no built-in Laravel queue worker. Never treat an un-awaited Promise
or a long Server Action as a durable job: a Vercel function can stop or timeout.

[ ] Learn three types of background work:
    1. `after()` from next/server for short, non-critical post-response work.
       It still shares the deployed route's time limit.
    2. Vercel Cron calling a protected Route Handler for scheduled work.
       Laravel comparison: scheduler/cron.
    3. A durable job service for retries, delays, and long work.
       Laravel comparison: Job classes + queue worker + failed_jobs.
[ ] Choose a durable queue:
    - Good Vercel learning choice: Inngest or Trigger.dev.
    - Self-hosted choice: BullMQ + Redis + a separate always-on worker.
    - Do not run an infinite worker inside the Vercel web app.
[ ] Create the first job: send an order-confirmation email.
[ ] Dispatch it after the Stripe webhook safely creates the order.
[ ] Add retries, exponential backoff, and failure logging.
[ ] Add an idempotency key so retries cannot duplicate important work.
[ ] Run Next.js and the chosen job dev worker in two terminals.
    -> Laravel: `php artisan serve` + `php artisan queue:work`.
[ ] Inspect queued/running/completed/failed jobs in the provider dashboard.
[ ] Add one scheduled cleanup job.

CHECKPOINT: Checkout does not wait for email; failed work retries visibly and
replaying a job does not duplicate database changes.
